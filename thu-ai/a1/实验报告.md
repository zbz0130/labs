## 2.语料库和数据预处理方法
语料库选用新浪新闻2016年的新闻语料库`语料库/sina_news_gbk`
数据预处理见data_process.py文件，其中包含两个函数
`load_materials()`:
读取gbk编码的语料库，将新闻的标题(title)和正文内容(html)合并，用换行符分离。将处理后的结果转存为UTF-8编码的materials.txt
`generate_ngram_count()`：
从生成的materials.txt中得到oj格式的一、二元词频表。具体实现为首先通过读取一二级汉字表得到合法汉字集合`valid_char`,之后逐行读取materials.txt,并用正则表达式`re.split(r'[^\u4e00-\u9fa5]+', line)`分块，将长文本按照非汉字字符切开，只保留纯汉字块，确保后续统计词频时不会出现跨标点统计的情况。在纯汉字块的基础上统计单个汉字出现的频次(Unigram)和连续两个汉字共同出现的频次(Bigram)。通过拼音汉字表得到从汉字到拼音的映射`hz_py`。因为一个汉字可能对应多个拼音（多音字），代码通过两层嵌套循环 for i in range(len(hz_py[...])) 确保了所有可能的拼音组合都被记录下来，生成与oj上数据同样格式的1_word.txt和2_word.txt。此外由于大部分二元词都无意义，只保留2_word.txt的频数前1e6个词，这样既加快数据生成时间，减小数据大小也不影响输出准确度。

## 3.基于字的二元模型的拼音输入法
__基本思路__：利用字和二元词的频数计算转移概率求使得P(S)=  $\prod_{i=1}^n P(w_i | w_{i-1})最大的句子$
**公式推导**:拼音输入法需要找P(S|O) = P(S)P(O|S)/P(O)最大的S
其中P(O)为常量，P(O|S)$\approx$1,因此只需要关注P(S)=$\prod_{i=1}^nP(w_i|w_1 \dots w_{i-1})$最大，等价于$\sum _{i=1}^nlog(P(w_i|w_1 \dots w_{i-1}))$最大。
在只关心二元语法时P(S) = $\prod_{i=1}^nP(w_i|w_{i-1})$
其中$P(w_i|w_{i-1}) = \begin{cases}
\frac{P(w_{i-1}w_i)}{P(w_{i-1})} = \frac{cnt(w_{i-1}w_i)}{cnt(w_{i-1})}  &\text{i > 0 } \\
P(w_0) = \frac{cnt(w_0)}{tot\_nums} &\text{i=0}
\end{cases}$
存在这样的$w_{i-1}w_i$频数为0，在求log时会发生错误，因此引入平滑处理
$P(w_i|w_{i-1}) = \begin{cases}
\lambda \frac{cnt(w_{i-1}w_i)}{cnt(w_{i-1})} + (1-\lambda)\frac{cnt(w_i)}{tot\_nums}&\text{i > 0 }  \\
P(w_0)  = \frac{cnt(w_0)}{tot\_nums}&\text{i=0}
\end{cases}$
**算法原理**
采用维特比算法（动态规划）,令dp[i][chr]为第i个拼音处，汉字为chr的最大概率，backptr[i][chr]为使得第i个拼音处汉字为chr概率最大时第i-1个拼音处的汉字，用于回溯结果。
转移方程$$\text{dp}[t][w_{curr}] = \max_{w_{prev}} (\text{dp}[t-1][w_{prev}] + \log P(w_{curr}|w_{prev}))$$
代码采用了**线性插值平滑**：

$$P(w_i|w_{i-1}) = \lambda \cdot \frac{Count(w_{i-1}, w_i)}{Count(w_{i-1})} + (1-\lambda) \cdot P(w_i)$$

*   **`LAMBDA` (0.99)**：平滑参数。如果二元词频存在，优先信任二元模型；如果不存在，则靠一元模型（$P(w_i)$）兜底。
*   **`SMALL` (0.001)**：极其微小的保底概率，防止因某种极端情况导致 `log(0)` 出错。
  * `backptr` 字典记录了当前最优路径是从上一个时刻的哪一个字跳过来的，用于最后生成句子。
 算法结束后，从最后一轮 `dp` 表中找到概率最大的汉字 `best_ch`，然后顺着 `backptr` 往回找，最后将结果翻转，得到最终输出。

#### 缺陷分析
* 由于采用新浪新闻作为语料库，拼音输入发会倾向于将词语识别成新闻中出现频率高的、更正式的词。如"青蛙"被识别成"青瓦"（新闻中更多出现青瓦台），"喵喵"被识别成"苗苗"
* 多音字导致词语识别错误，由于在先前的数据预处理阶段粗糙的把一个词加到所有对应的拼音组合中(如给予会加到"gei yu"和"ji yu")，这回导致词语识别错误如"教学效果"被识别成"校学校国"
* 模型只关注相邻词的关系而不关心整句结构，目光短浅。如"很多属于"会由于"多数"的频数高而被识别成"很多**数与**",“北京是首个”被识别成"北京**市**首个"

### 4.思考题
#### 1.语料编码
**gbk和utf-8的区别**
| 特性 | **GBK** (汉字内码扩展规范) | **UTF-8** (万国码变长实现) |
| :--- | :--- | :--- |
| **标准背景** | 中国国家标准，专门为简体中文设计。 | 国际通用标准，旨在统一全球所有语言。 |
| **编码长度** | **变长（1-2字节）**：英文字符 1 字节，汉字固定 **2 字节**。 | **变长（1-4字节）**：英文字符 1 字节，汉字通常占 **3 字节**。 |
| **兼容性** | 向上兼容 ASCII；但不兼容非中文字符集（如韩文、日文会乱码）。 | 向上兼容 ASCII；**全球通用**，在 Linux、Web、移动端是标准。 |
| **存储效率** | 存储**纯中文**文档时，体积比 UTF-8 小约 33%。 | 存储中文比 GBK 费空间，但处理多语言混合文本时极具优势。 |
| **应用场景** | Windows 简体中文版系统内部、旧版政府/银行软件、部分中文网页。 | 现代编程（Python/C++）、Linux/WSL 系统、绝大多数互联网应用。 |

存在其他编码方式

(1) ASCII (American Standard Code for Information Interchange)
*   **说明**：最古老的标准，仅使用 7 位二进制，包含 128 个字符（英文字母、数字和控制符号）。
*   **应用场景**：计算机底层协议、编程语言的关键字、简单的纯英文配置文件。它是所有现代编码（GBK, UTF-8）的基石。

(2) UTF-16
*   **说明**：Unicode 的另一种实现，通常使用固定 **2 字节**（或 4 字节）表示一个字符。
*   **应用场景**：**Windows 操作系统的内部运行机制**、Java 和 .NET 语言的内存字符串处理（String 对象在内存中通常以 UTF-16 存储）。

(3) Big5 (大五码)
*   **说明**：专门为**繁体中文**设计的编码标准。
*   **应用场景**：主要用于中国台湾、香港、澳门地区的旧有系统和网页。如果你处理来自这些地区的语料，遇到乱码通常需要尝试用 Big5 解码。

(4) GB18030
*   **说明**：GBK 的升级版，也是目前中国强制执行的标准。它向下兼容 GBK，但支持更多的少数民族文字（如藏文、维文）和 Emoji。
*   **应用场景**：我国政府机构强制要求的软件版本、需要支持极生僻字的大型数据库。

 (5) ISO-8859-1 (Latin-1)
*   **说明**：单字节编码，涵盖了大多数西欧语言（德语、法语、西班牙语等）。
*   **应用场景**：早期的 HTTP 传输默认编码。在处理一些旧的 Web 数据时，如果中文显示为“Ã©”之类的符号，通常是因为被错误地按 Latin-1 解码了。
#### 2.算法设计
假设输⼊句⼦由n个字组成，字库⼤⼩为V并且在读音上均匀分布，读音数量有限为常数C 。
字在读音上均匀分布意味着每个拼音对应的候选汉字数量大约为 $K = \frac{V}{C}$。
**时间复杂度分析** 
记K(状态)=当前拼音的候选汉字数


1.  **处理第一个字**：
    计算初始概率，复杂度为 $O(K)$。
2.  **递推步骤（从第 2 个字到第 $n$ 个字）**：
  对于每一个时刻 $t$（共 $n-1$ 个时刻）：
    *   我们需要计算到达当前时刻 $K$ 个候选状态中每一个状态的最优路径。
    *   为了确定到达当前某个状态（汉字）的最优路径，需要遍历前一时刻的所有 $K$ 个状态。
    *   **计算量**：$(n-1) \times K (\text{当前状态}) \times K (\text{前一时刻状态}) = O(n \cdot K^2)$。

    将 $K = \frac{V}{C}$ 代入：
    $$T = O\left(n \cdot \left(\frac{V}{C}\right)^2\right)$$
    其中$\frac{1}{C^2}$ 是常数。

**结论**：时间复杂度为 **$O(n \cdot V^2)$**。

---

空间复杂度分析 
空间复杂度主要取决于我们需要存储的数据结构：
1.  **DP 表**：
    存储每个时刻、每个候选状态的最大对数概率。
    大小为 $n \times K$。
2.  **回溯指针表**：
    存储每个时刻、每个候选状态是由前一时刻的哪一个状态转移过来的，以便最后回溯路径。
    大小为 $n \times K$。
3.  **代入参数**：
    $$S = O(n \cdot K) = O\left(n \cdot \frac{V}{C}\right)$$
    同样地，$C$ 为常数。

**结论**：空间复杂度为 **$O(n \cdot V)$**。

#### 3.输入输出
* 代码A是一种即时处理的在线算法，采用边读边写的模式。每从标准输入读入一行数据就立刻向标准输出写入一行。在终端中输入和输出是交替出现的。
* 代码B是一种采用缓存处理的离线算法，采用先读后写的模式。先将所有输入数据读取并存储在列表l中，直到检测到输入结束(EOF)才将所有数据输出。在终端中输入和输出是分离开来的。
* 代码A的空间复杂度为O(1),代码B的空间复杂度为O(n)
* 两份代码都是正确的，不过代码A更好，其内存消耗小，更不容易MLE，且在交互式题目时必须使用代码A,代码B会导致程序死锁。

#### 实现三元模型
__基本思路__:

三元模型的概率公式为：
$$P(W) \approx P(w_1) \cdot P(w_2|w_1) \cdot P(w_3|w_1, w_2) \dots P(w_n|w_{n-2}, w_{n-1})$$

由于三元词组在语料库中极其稀疏（很多三个字的组合从未出现过）,依然采用线性插值平滑
$$P(w_i|w_{i-2}, w_{i-1}) = \lambda_3 \frac{Count(w_{i-2}, w_{i-1}, w_i)}{Count(w_{i-2}, w_{i-1})} + \lambda_2 \frac{Count(w_{i-1}, w_i)}{Count(w_{i-1})} + \lambda_1 P(w_i)$$
其中 $\lambda_3 + \lambda_2 + \lambda_1 = 1$（此处先取 0.7, 0.2, 0.1）。

`dp[i][(w_prev, w_curr)] = max_log_prob`

三元dp的状态不再是单个汉字，而是汉字对。之后的算法跟二元情况相似，依据上面的公式进行状态转移，最后找到整个字符串概率最大的路径，再根据backptr回溯出答案。

#### 二元模型$\lambda$值

|$\lambda$|0.9|0.99|0.999|1.0|
|:---|:---|:---|:---|:---|
|句准确率|41.92%|42.12%|41.12%|17.96%|
|字准确率|85.56%|84.89%|84.74%|70.32%|

最终选择$\lambda$=0.99
$\lambda$会决定模型关注二元词组的程度，为了保证模型识别出常用的二元词组，$\lambda$需要保持一个较高的值(0.9以上)。然而$\lambda$过高则会使模型错误的关注不是词组的二元字组合从而降低准确率。

在三元模型中，为了加快推理速度，我们采用了剪枝的思想。剪枝有两个参数
```
MAX_CANDIDATES ：每个拼音最多只考虑一元词频最高的前 MAX_CANDIDATES 个汉字
BEAM_SIZE :每一轮最多只保留概率最大的 BEAM_SIZE 条路径状态
```

## 7.其他评价指标




#### **1.交互效率指标**
这类指标衡量用户为了输入正确文字所付出的“体力”成本。

*   **KSPC (Keystrokes Per Character，单字平均按键数)**：
    *   **定义**：用户输入一段文字总共敲击键盘的次数除以最终生成的汉字个数。
    *   **意义**：KSPC 越低，说明输入法越聪明（例如支持简拼、联想和自动补全）。如果模型能通过 `py` 就猜出 `拼音`，KSPC 就会显著下降。
*   **前 N 项命中率 (Top-N Hit Rate)**：
    *   **定义**：正确候选项出现在选词列表前 N 位（通常看 Top-1, Top-3, Top-5）的频率。
    *   **意义**：如果 Top-1 命中率高，用户可以盲打（直接空格）；如果命中率在 Top-5 之外，用户需要频繁翻页或手动选择，交互体验会极差。
*   **翻页率 (Page-down Rate)**：
    *   **定义**：用户在输入过程中需要点击“下一页”寻找候选项的比例。
    *   **意义**：反映了模型排序算法的质量。

分析样例输出中出现的错误句子也很有趣，发现了很多初级输入法的bug，其中有些问题商业化成功的搜狗输入法也拥有。根据发现的问题改进模型和调整参数的过程也其乐无穷。
**有趣的发现**
我自己发现了一个实现拼音输入法屏蔽违禁词的方法：
可以在n_word.txt中将想要屏蔽的词的词频设置为负数，在viterbi算法中我们计算的是对数概率，频率是负数的违禁词取对数会报错，这时加try和except即可实现屏蔽违禁词功能
