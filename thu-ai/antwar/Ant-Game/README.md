## 一、游戏规则简介

​这是一款双人在六边形网格地图上对抗的策略游戏，双方玩家分别操控己方基地，通过每回合自动生成并派遣蚂蚁进攻敌方基地，同时可以消耗金币建造、升级或降级多种类型的防御塔来防守和干扰敌人，也能使用四种具有不同持续效果和冷却时间的超级武器（如闪电风暴、EMP轰炸、引力护盾和紧急回避）来改变局部战局。

## 二、游戏规则详解

### 1. 基本数据

- 本游戏回合数从**0**开始，双方分别操作一次算作一回合。回合数到达512时结束游戏。
- 蚂蚁具有年龄。年龄等于**当前回合数 - 生成回合数**。普通蚂蚁当年龄大于64时会死亡/消失；战斗蚂蚁没有寿命上限，不会仅因存活过久而死亡。
- 蚂蚁具有两个属性：
  - **兵种(kind)**：**普通蚂蚁**、**战斗蚂蚁**。
  - **行为(behavior)**：**保守型**、**默认型**、**随机型**、**蛊惑型**、**免控型**。
- 行为决定寻路和受控方式；兵种决定移动权重、拆塔能力与特殊体质（详见2.7节）。
- 规定AI每回合运行时间上限为**10秒**。

每个格点都有六个方向，计算相邻坐标的算法如下所示。需要额外注意的就是本坐标系在奇数列和偶数列需要分别处理。

六个下标的顺序为右上、上、左上、左下、下、右下。

```python
direction_difference = [[
    # even columns
    [0,  1], [-1, 0], [0, -1],
    [1, -1], [ 1, 0], [1,  1],
],[
    # odd columns
    [-1,  1], [-1,  0], [-1, -1],
    [ 0, -1], [ 1,  0], [ 0,  1],
]]

def calculate_neighbor(pos, direction):
    diff = direction_difference[place % 2][direction]
    return [pos[0] + diff[0], pos[1] + diff[1]]
```

### 3. 经济系统
- 玩家初始具有50金币。
- 每两个回合结算一次基础收入，每次给予双方玩家3金币。
- **在结算阶段**，本回合每使用防御塔或超级武器杀死一只对方蚂蚁时，普通蚂蚁按照等级给金币：等级1给予6金币，等级2给予10金币，等级3给予14金币；战斗蚂蚁的击杀收益固定为18金币。本回合每有一只蚂蚁到达敌方基地，给予10金币。因寿命死亡的蚂蚁不对金币产生影响。

### 4. 防御塔

- **建造价格**：建造**新的**防御塔的价格为 $15\times 2^i$，其中i为当前存在的己方防御塔的数量。或者说，建造第一个防御塔的金额为15，建造第二座防御塔的价格为30，建造第三座防御塔的价格为60，以后每座防御塔的建造价格均翻倍。

- **升级价格**：防御塔一共分为三个等级，共13种类。等级1升级到等级2需要60金币。

- **塔生命值变化**：`Basic` 防御塔最大血量为 `10`，其余高级防御塔最大血量均为 `15`。防御塔**升级时会回满血**；防御塔**降级时按血量比例继承**，即新血量为 $\left\lceil \text{新最大血量}\times\frac{\text{当前血量}}{\text{原最大血量}} \right\rceil$。

- **降级、拆除返还**：返还统一按“**对应价格** $\times 90\% \times \frac{\text{当前血量}}{\text{最大血量}}$”并**向下取整**计算。对非 `Basic` 塔执行降级时，“对应价格”指本次降级所退回的升级花费（等级1降回 `Basic` 以 `60` 为基数，等级2降回等级1以 `200` 为基数）；拆除 `Basic` 塔时，“对应价格”指 $15\times 2^i$，其中 $i$ 为**拆除之后**己方防御塔的数量。

- **防御塔攻击**：每种类防御塔都有固定的伤害、攻击间隔、攻击次数、攻击方式、攻击范围。*攻击间隔为K，攻击次数为M*意为防御塔内置CD为0时，可以进行M次攻击，若攻击到任何“工蚁“，则重置CD为K，否则CD不变。

  还需指出：建造、升级、降级防御塔都会令内置CD为攻击间隔K。

- **索敌逻辑**：选取范围内血量大于0且非己方的敌人，按照与防御塔的距离为主键、“工蚁”的ID为附键升序排列。顺序越靠前攻击优先级越高。用代码表示为：

  ```python
  targets = ants_in_range(tower.x, tower.y, tower.range)
              .filter((ant) => ant.hp > 0 && ant.player != tower.player)
              .sort((a, b) => a_dist == b_dist ? a.id - b.id : a_dist - b_dist)
  // a_dist = distance([tower.x, tower.y], [ant.x, ant.y])
  ```

- 若非特殊说明，默认的攻击方式就是对优先级最高的“工蚁”造成一定的伤害。

- **AOE攻击**：部分防御塔拥有“AOE”(Area of effect)攻击属性。若为“范围为R的AOE攻击”，即为对优先级最高的“工蚁”及到其所在格距离不大于R的格子中所有“工蚁”均造成伤害。**这有可能影响到本来不处于攻击范围内的敌方“工蚁”。**

### 6. 基地

- 每方基地在游戏开始时有50血量，每当一只敌方“工蚁”移动到基地，则会减少1血量。当有一方基地血量不大于0时游戏立即结束。
- 基地可以进行如下升级：
- **优化生产流水线**：基地会按下表对应的生产周期自动建造一只“工蚁”。初始为1级；当周期为 `4.5` 时，会按 `0, 5, 9, 14, 18, ...` 这样的离散节奏产兵；当周期为 `3.5` 时，会按 `0, 4, 7, 11, 14, ...` 这样的离散节奏产兵。

|      | 1级  | 2级  | 3级  |
| ---- | ---- | ---- | ---- |
| $K$  | 4.5  | 4    | 3.5  |

- **列装高级护甲**：产生“工蚁”机器人的最大生命值。注意这一属性升级时，已产生的“工蚁”最大生命值**不会改变**。初始为1级。

|          | 1级  | 2级  | 3级  |
| -------- | ---- | ---- | ---- |
| 最大生命 | 20   | 25   | 25   |

- **花费**：1级升级为2级：200。2级升级为3级：250。无法降级。
- 一回合只能对基地进行一种升级。

### 7. “工蚁”寻路算法

蚂蚁现在同时拥有**兵种(kind)**与**行为(behavior)**两个维度。行为决定移动方式，兵种决定移动画像与拆塔能力。当前比赛默认运行时使用增强型寻路策略 `enhanced`；仓库中仍保留旧版局部打分策略 `legacy` 用于回归测试与兼容。两套策略共享同一套基础约束与风险场定义。所有蚂蚁在移动时都遵循以下基本约束：

- 可以选择的目标格包括：普通路径格、敌方基地、以及**相邻敌方塔所在格**。
- 若选择的是相邻敌方塔所在格，则本回合结算为**攻击塔**：蚂蚁实际位置不变化，通常只结算对该塔的伤害；若触发战斗蚂蚁自爆规则，则按自爆规则结算。
- 默认情况下不能走**回头路**：本回合不能移动到上一回合所在的位置。首次移动、传送后，以及对塔执行“虚假移动攻击”后，回头路限制会被重置。
- **随机型**与**蛊惑型**允许回头；其他行为默认不允许。若非回头候选为空，则会退化为允许回头重新选方向。

#### 兵种

- **普通蚂蚁**：常规进攻单位，拆塔伤害按等级分别为 `1 / 2 / 4`。
- **战斗蚂蚁**：更偏向寻塔拆塔，拆塔伤害固定为 `5`，最大生命固定为 `30`，出生时自带 `3` 层**紧急回避**，且**没有寿命上限**；若当前生命值低于自身最大生命值一半，则攻击塔时改为自爆：对目标塔及其距离 `1` 内的敌方防御塔各造成 `10` 点伤害，然后自身死亡。基地与产蚁塔会以较高概率自然生成为战斗蚂蚁，`ProducerSiege` 还会额外生成战斗蚂蚁。

#### 行为与生成概率

基地与产蚁塔生成的新蚂蚁，其出生结果按以下概率随机确定：

- **默认型普通蚂蚁**：40%
- **保守型普通蚂蚁**：35%
- **随机型普通蚂蚁**：10%
- **默认型战斗蚁**：15%

注：

- **蛊惑型**不会自然生成，只能由控制效果转化而来。
- **免控型**不会自然生成；只能由技能、控制免疫退化规则等效果转化而来。
- 免控型蚂蚁免疫冻结、行为转化等控制效果，且不计入每10回合的随机移动选择中。

#### 行为特性

- **默认型**：使用风险感知寻路。在默认的 `enhanced` 策略下，会先根据全局路径代价做预规划，再在候选方向间结合信息素等局部因素做 softmax 采样；在 `legacy` 策略下，则直接对本回合候选方向做局部评分并采样。当前默认温度参数为 `1.75`。
- **保守型**：与默认型共享同一套寻路框架，但不采样，而是直接选择分数最高的方向。进入保守型后的第5回合结束时退化为**默认型**。
- **随机型**：在合法候选方向中均匀随机选择。随机型在进入后的第5回合结束时退化为**默认型**。
- **蛊惑型**：目标不再是敌方基地，而是由 `Bewitch` 指定的内部目标格；其余仍使用风险感知评分，但更偏向朝该目标推进，并允许回头。当前实现中，蛊惑型无论开关 `enhanced / legacy`，都沿用这套局部目标评分逻辑。若提前抵达目标格，或持续5回合结束，则退化为**默认型**。
- **免控型**：移动规则与保守型一致，但免疫控制。进入免控型后的第5回合结束时退化为**默认型**。

#### 风险感知寻路

当前实现会先维护三类静态场：

- `damage`：估计该格在本回合会承受的伤害强度。除敌方防御塔外，敌方 `Lightning Storm` 当前覆盖区域也会额外写入该场，因此蚂蚁会主动规避闪电风暴区域。
- `control`：估计该格被 `Ice / Bewitch / Pulse` 命中的控制风险。
- `effectPull`：估计该格受到的友方支援吸引。当前会把友方 `Deflectors` 与 `Emergency Evasion` 的覆盖区域写入该场，因此蚂蚁会更愿意靠近这些区域。`EMP Blaster` 不会直接写入寻路场，而是通过禁用塔攻击来间接改变局面。

在 `legacy` 策略下，对默认型、保守型、免控型以及蛊惑型蚂蚁，当前移动评分可概括为：

$$
\mathrm{score} = w_p \cdot progress + w_\tau \cdot pheromone - w_c \cdot crowding - w_d \cdot damage - w_r \cdot control + w_t \cdot towerPull + w_e \cdot effectPull
$$

其中：

- `progress` 以“更接近目标”为主，但会额外惩罚停滞和后退，并加入一个随距离衰减的敌方主塔吸引项。
- `pheromone` 为己方信息素。
- `crowding` 会惩罚与己方蚂蚁过于拥挤的落点。
- `damage` 估计该格会受到的塔伤害强度。
- `control` 估计该格被 `Ice / Bewitch / Pulse` 命中的控制风险。
- `towerPull` 仅对战斗蚂蚁明显有效，用于鼓励其接近敌塔、优先收残塔。
- `effectPull` 表示友方支援区域对路径的吸引度，目前主要来自 `Deflectors / Emergency Evasion`。

在默认的 `enhanced` 策略下，寻路不再只看当前一步的局部评分，而是先做一轮反向加权规划：

$$
\mathrm{stepCost} = \max(\varepsilon, 1 + \alpha \cdot damage + \beta \cdot control + \gamma \cdot traffic - \delta \cdot effectPull)
$$

其中 `traffic` 表示己方同回合已预留或预测会拥挤到该格的程度，`effectPull` 则会降低步进代价，从而把路径拉向友方 `Deflectors / Emergency Evasion` 区域。其核心特征如下：

- **普通蚂蚁**：以敌方基地为目标，对全图预先计算总代价；落点会额外考虑拥塞预留、信息素以及“若继续前进收益更高，则尽量不要为了顺手攻击塔而停下”。
- **战斗蚂蚁**：会同时评估“直接走向敌方基地”和“接近并攻击某座敌塔”的收益。若邻格可直接攻击敌塔，则会根据直伤、自爆收益、补刀收益和剩余生命值决定是否出手；否则会选择对某座敌塔或敌方基地综合收益最高的路线。
- **同回合分流**：增强型寻路在移动阶段会记录已选择落点和已声明攻击的塔，后续蚂蚁会对这些格子和塔目标施加额外惩罚，从而减少扎堆走同一路或过度集火同一座塔的情况。

#### 信息素

信息素会影响移动。

- **初始值**：每格信息素由随机种子生成，真实值在 `[8.0, 12.0)` 区间内，双方独立。
- **内部表示**：游戏运行时为了保持 C++ / Python 端的整数定点一致性，会按 `10000` 倍存储信息素；也就是实际值 `10.0` 在内部会记成 `100000`。参与移动评分、导出给人类查看时都会再除回 `10000`，因此原始矩阵里看到 `8xxxx ~ 12xxxx` 属于正常现象，不是实际值直接变成了几万。
- **更新**：
  - 当有蚂蚁**攻入敌方基地**，其路径上的点获得 `+10`
  - 当有蚂蚁因**生命值耗尽**死亡，其路径上的点获得 `-5`
  - 当有蚂蚁因**年龄过大**死亡，其路径上的点获得 `-3`
  - 同一只蚂蚁对同一个点在单次结算中只更新一次
  - 每回合还会做一次全局衰减，近似公式为：$\tau'_P \approx 0.97 \tau_P + 0.3 + \sum_k \Delta\tau_P^{(k)}$

#### 推进偏好与回头路限制

- `progress` 项仍然鼓励蚂蚁接近当前目标，但现在不是简单的 `1.25 / 1.00 / 0.75` 三档吸引度，而是直接按距离变化、停滞惩罚、后退惩罚和敌方主塔吸引项共同计算。
- 所有非随机型、非蛊惑型蚂蚁默认受回头路限制；若没有其他合法方向，才会允许回头。

#### 随机移动

游戏每10回合进行一次随机移动阶段。候选目标为场上所有**非免控型**且未死亡、未老死的蚂蚁，其中随机选取10%（至少1只）参与结算。选中后按选中顺序逐只结算，每只连续结算3步：

- 每一步有 `2/3` 概率在当前**合法候选方向**中均匀随机选择一个方向。
- 每一步有 `1/3` 概率重新调用该蚂蚁此刻的移动算法来选择方向。
- 这3步是**一步一步顺序结算**的：前一步结算完后，才会基于更新后的站位、拥塞、塔状态等信息结算下一步。
- 若所选方向对应相邻敌塔，则该步结算为攻击塔而非真实位移。

### 8. 胜负判定

- 大本营剩余血量多者，胜。如果剩余血量相同：
- 击败对方蚂蚁数多者，胜。如果击败蚂蚁数相同：
- 使用超级武器少者，胜。如果使用超级武器次数相同：
- AI用时少者，胜。如果用时相同：
- 先手胜。

### 9. 结算流程

1. 玩家操作
   1. 等待玩家0的操作，若玩家0程序崩溃、运行超时、返回了不符合协议的数据、执行了非法的操作，那么立刻判负。
   2. 执行玩家0的操作，如建造、升级防御塔，升级基地、使用超级武器等；若本次操作部署了超级武器，则该武器的**即时效果当场结算**。
   3. 然后对玩家1执行同样的步骤
2. 回合结算
   1. **处理攻击前状态**：解除冻结状态并恢复待执行行为；同时更新引力护盾、紧急回避等会在攻击阶段生效的状态。
   2. 结算闪电风暴：对所有处于闪电风暴**当前区域**内的敌方蚂蚁造成20伤害；若为该效果生效的第5、10、15回合，再对范围内的敌方防御塔造成3点伤害。
   3. 按照建造顺序结算防御塔攻击。攻击时首先判断塔是否处于EMP干扰**当前区域**内（若是，则该防御塔无法攻击）；然后执行伤害和附加效果。生产塔分支不会主动攻击。
   4. 若本回合造成敌方”工蚁“死亡（生命值变为非正数），则按照死亡的”工蚁“的等级获得金币。
   5. 如果**普通蚂蚁**的年龄大于最大年龄，即**当前回合数-蚂蚁生成回合数大于64**，标记为老死；战斗蚂蚁不会因年龄过大而死亡。
   6. 按照生成顺序结算蚂蚁的常规移动。
      1. 已死亡的蚂蚁不可以移动。
      2. 被冻结的蚂蚁这回合无法移动，并在解冻后按规则切回待执行行为；若冻结由Ice防御塔施加，则解冻后会转为随机型。
      3. 其余存活蚂蚁按自身行为和兵种画像选择方向；若目标是相邻敌塔，则本回合结算为攻击塔而非真实位移。
      4. 如果蚂蚁真实移动到敌方基地，则立刻减少敌方基地血量（-1），并判断游戏是否结束。
   7. **随机移动处理**：若当前回合数满足每10回合的条件，执行随机移动阶段（见2.7节）。
   8. 结算信息素并移除已死亡、已到达的蚂蚁；到达的蚂蚁结算金币收益（每只10金）。
   9. 按照基地等级尝试生成普通蚂蚁；随后结算产蚁塔分支的额外产兵与治疗/支援效果。
   10. 所有存活蚂蚁年龄+1；同时处理行为计时：随机型按进入随机型后的回合数退化，保守型/蛊惑型/免控型按各自独立剩余回合过期并退化为默认型。
   11. 若本回合结算后凑满两个回合，则双方金币+3
   12. 结算超级武器持续时间与冷却时间：对于正在生效的闪电风暴和EMP轰炸，其区域中心会在**当前位置**与相邻合法位置中随机选择一格，作为下一回合的位置；随后结算持续时间与冷却时间。
   13. 回合数+1
   14. 若回合数等于512，执行胜负判断，结束游戏

## 三、选手AI编写指南

### 1. 输入输出

选手AI可以从标准输入流**直接读取**来自评测系统的信息。

但是，选手AI通过标准输入流向评测系统返回信息时，需要在发送的信息之前添加一个**四字节大端序**整数，代表信息的长度。如选手想要输出以下信息：

```c++
2
1 2 3 4 5
```

这条信息的长度为11，其十六进制数据表示为（其中第二行的`-`为空格）：

```bash
HEX : 32 0A 31 20 32 20 33 20 34 20 35
TEXT:  1 \n  1  _  2  _  3  _  4  _  5
```

因此需要在数据包前面加上11的四字节大端序表示`00 00 00 0B`，最终结果为：

```bash
HEX : 00 00 00 0B 32 0A 31 20 32 20 33 20 34 20 35
TEXT:  0  0  0 11  1 \n  1  _  2  _  3  _  4  _  5
```

选手AI可以通过向标准错误流输出信息进行调试，在Saiblo平台上评测完成后，会提供标准错误流产生的信息。请注意最终提交时请尽量减少调试信息的输出，因为这可能会占用大量运行时间。

### 2. 评测流程

- 总体评测流程如下
  - 平台启动双方玩家AI程序，并向双方玩家发送初始化信息
  - 每一回合均按照以下流程顺序、循环执行
    1. 等待先手玩家操作
    2. 平台接收到先手玩家操作后，进行验证和执行，如果成功完成，将操作转发给后手玩家
    3. 等待后手玩家操作
    4. 平台接收到后手玩家操作后，进行验证和执行，如果成功完成，将操作转发给先手玩家
    5. 游戏逻辑进行一回合的结算，如果回合正常结束，则将局面信息通过平台分别发送给两位玩家。如果游戏结束，那么会向平台汇报游戏结果，终止评测流程。
- 对于**先手玩家**的AI，执行流程如下：
  - 接受初始化信息，判断自己是先手玩家
  - 每一回合中：
    1. 进行决策，发送自己的操作
    2. 等待接受后手玩家的操作
    3. 等待接受局面信息
- 对于**后手玩家**的AI，执行流程如下：
  - 接受初始化信息，判断自己是后手玩家
  - 每一回合中：
    1. 等待接受先手玩家的操作
    2. 进行决策，发送自己的操作
    3. 等待接受局面信息

### 3. 游戏初始化信息

一行两个整数`K M`，`K == 0`代表自己为先手玩家P0，`K == 1`代表自己为后手玩家P1；`M`为随机数种子，用于计算初始局面的信息素，计算算法如下：

```c++
// C++ Version

unsigned long long lcg_seed;
unsigned long long lcg(){
    lcg_seed = (25214903917 * lcg_seed) & ((1ll << 48) - 1);
    return lcg_seed;
}

void init_pheromon(unsigned long long M){
    lcg_seed = M;
    for(int i = 0; i < 2; i++)
        for(int j = 0; j < MAP_SIZE; j++)
            for(int k = 0; k < MAP_SIZE; k++)
                pheromone[i][j][k] = lcg() * pow(2, -46) + 8;
}
```

```python
# Python Version

lcg_seed = 0
def lcg():
    global lcg_seed
    lcg_seed = (25214903917 * lcg_seed) & ((1 << 48) - 1)
    return lcg_seed

def init_pheromon(M):
    global lcg_seed
    lcg_seed = M
    for i in [0,1]:
        for j in range(0, MAP_SIZE):
            for k in range(0, MAP_SIZE):
                pheromone[i][j][k] = lcg() * pow(2, -46) + 8
```

实际运行时，原生 C++ 引擎和 `SDK/backend/engine.py` 都会把这份真实值按 `10000` 倍转成定点整数后再存储，以保证跨语言的确定性和一致的舍入行为；因此内部矩阵会表现为 `80000 ~ 120000`，对应真实值仍然是 `8.0000 ~ 12.0000`。

### 4. 玩家操作信息

第一行一个整数$N$，代表操作数。接下来$N$行，每行表示一个操作，第一个整数$T$代表操作类型，后面需要根据操作类型提供一些参数。操作类型和参数如下：

- 建造防御塔：
  - 格式：`11 x y` 在`(x, y)`处建造防御塔
  - 例子：`11 11 1` 在`(11, 1)`处建立一座新的防御塔
- 升级防御塔：
  - 格式：`12 towerId towerTypeId` 将ID为`towerId`的防御塔升级为`towerTypeId`所代表的防御塔类型
  - 例子：`12 1 31` 将ID为`1`的防御塔升级为`31`类型，也即`Mortar+` 
- 降级防御塔：
  - 格式：`13 towerId` 降级ID为`towerId`的防御塔。如果已经是`Basic`防御塔，那么会将其**拆除**。
  - 例子：`13 1` 降级ID为`1`的防御塔。如果已经是`Basic`防御塔，那么会将其**拆除**。
- 超级武器：
  - 格式：`21 x y` `22 x y` `23 x y` `24 x y` 分别在`(x, y)`处部署ID为1/2/3/4的超级武器。
  - 例子：`21 9 9` 在`(9, 9)`的位置释放“闪电风暴”
- 基地升级：
  - 格式：`31` 代表升级“生产流水线（出兵速度）”。`32`升级“高级装甲（最大生命）”。
  - 例子：`31` 升级“生产流水线（出兵速度）”

可能的非法操作/限制包括。

如下是一个完整的操作信息的示例：

```c++
5
11 11 1
12 1 31
13 2
21 9 9
31
```

### 5. 局面信息

第一行一个整数`R`，表示回合数。

第二行一个整数`N_1`，代表场上总防御塔数量。接下来`N_1`行，每行**7**个整数，分别为`id player x y type cd hp`，即每个防御塔的ID、归属、坐标、类型、攻击CD、当前生命。

接下来一个整数`N_2`，代表场上总蚂蚁数量。接下来`N_2`行，每行**10**个整数，分别为`id player x y hp lv age state behavior kind`，即每只蚂蚁的ID、阵营、坐标、当前生命、等级、当前寿命、当前状态、当前行为、兵种。

- 蚂蚁等级分别为：`0 / 1 / 2`
- 当前状态 `state`：`0-4 = Alive / Success / Fail / TooOld / Frozen`
- 当前行为 `behavior`：`0-4 = Default / Conservative / Random / Bewitched / ControlFree`
- 当前兵种 `kind`：`0-1 = Worker / Combat`

接下来一行**2**个整数：`coin0 coin1`，表示双方当前金币。

接下来一行**6**个整数：`camp0_hp camp1_hp speedLv0 speedLv1 anthpLv0 anthpLv1`。

- `speedLv` 表示基地“生产流水线 / 出兵速度”升级线的内部等级，取值为 `0 / 1 / 2`，分别对应文档中的 `1 / 2 / 3` 级。
- `anthpLv` 表示基地“高级护甲 / 新生普通蚂蚁最大生命”升级线的内部等级，取值为 `0 / 1 / 2`，分别对应文档中的 `1 / 2 / 3` 级。

接下来一个整数`N_3`，代表超级武器冷却信息的行数。接下来`N_3`行，每行按武器ID `1-4` 的顺序给出一名玩家的冷却：`lightning_cd emp_cd deflector_cd emergency_evasion_cd`。

接下来一个整数`N_4`，代表当前仍在生效的超级武器区域数量。接下来`N_4`行，每行**5**个整数：`type player x y remaining_turns`，表示一个仍在生效的超级武器区域的类型、归属、中心坐标与剩余持续回合数。

#### 回放文件（`replay.json`）说明

评测器初始化参数中的 `replay` 路径，以及本地脚本 [tools/run_local_match.py](./tools/run_local_match.py) 产出的 `replay.json`，顶层都是一个 JSON 数组。数组下标就是回合号（从 `0` 开始），每个元素代表该回合双方操作执行完毕、并完成结算后的记录。

单个回合记录的典型结构如下：

```json
{
  "op0": [...],
  "op1": [...],
  "round_state": {...},
  "seed": 7
}
```

- `seed` 只会出现在第 `0` 回合的记录里，用于复现随机过程。
- `op0` / `op1` 分别是玩家 0 / 玩家 1 该回合的操作列表，字段格式与操作协议一致；未使用的字段通常记为 `-1`。
- `round_state` 是这一回合写入回放的局面结果。回放文件中会保留 `coins`、`camps`、`speedLv`、`anthpLv`、`weaponCooldowns`、`activeEffects`、`ants`、`towers`、`winner`、`message` 以及 `pheromone` 等字段。

其中：

- `round_state.ants` 中的每条记录包含 `player`、`id`、`pos`、`hp`、`level`、`age`、`status`、`behavior`、`kind`、`move`。
- `move = -1` 表示该蚂蚁本回合没有可记录的移动方向，例如刚生成。
- `round_state.towers` 不是“当前场上全部塔的完整快照”，而是“本回合发生变化的塔的增量记录”；每条记录额外包含当前 `hp`。

也就是说，如果你要从回放还原完整的防御塔状态，需要自己维护一个 `tower_id -> tower` 的映射，并按回合应用 `round_state.towers` 中的变化：

- `type >= 0`：表示该塔本回合被新建、升级、降级，或者发生了攻击；用这条记录更新本地状态即可。
- `type == -1`：表示该塔在本回合被真正拆除，应当从本地状态里删除。

这里的 `tower.type = -1` 不是任何合法的防御塔种类。合法防御塔类型只有：

```text
0, 1, 2, 3, 4, 11, 12, 13, 21, 22, 23, 31, 32, 33, 41, 42, 43
```

回放中的 `-1` 是输出层专门定义的“塔被拆除”标记。举例来说，下面这条记录表示 `id = 2` 的塔在该回合被移除了：

```json
{
  "cd": 1,
  "id": 2,
  "player": 0,
  "pos": {"x": 7, "y": 5},
  "type": -1
}
```

需要额外注意：

- 只有当 `Basic` 塔被 `13 id` 操作真正拆除，或塔在回合中被蚂蚁摧毁时，回放里才会写出 `type = -1`。
- 如果对非 `Basic` 塔执行 `13 id`，游戏行为是“降级”而不是“直接移除”，因此回放里会写入降级后的真实塔类型，而不是 `-1`。
- 塔记录里的 `attack` 字段只会在该塔本回合实际攻击、且 `type != -1` 时出现，内容是本回合命中的蚂蚁 `id` 列表。

### 6. SDK使用方法介绍

本仓库把“规则逻辑”“策略逻辑”“训练脚手架”拆成了三层：

- `SDK/backend/`：唯一的 Python 后端实现，负责游戏状态、规则推进、公共状态同步。
- `SDK/training/`：训练环境与训练基类，负责 self-play、观测、动作空间和训练循环。
- `AI/`：选手自己的策略代码，只负责根据当前状态选择动作，不应重复实现游戏规则。

#### 6.1 编写一个 AI

最简单的入口可以参考 [AI/ai_example.py](./AI/ai_example.py)。

你需要实现 `AI.choose_bundle()`，根据当前状态和候选动作选择一个 `ActionBundle`。

```python
from SDK.backend import BackendState
from SDK.utils.actions import ActionBundle

class AI(...):
    def choose_bundle(
        self,
        state: BackendState,
        player: int,
        bundles: list[ActionBundle] | None = None,
    ) -> ActionBundle:
        ...
```

推荐遵循下面的分工：

- 规则、合法性判断、回合推进：交给 `SDK.backend`
- 候选动作生成：交给 `SDK.utils.actions.ActionCatalog`
- 你的策略、打分、搜索：写在 `AI/*.py`

不要在 `AI/` 里再实现一套自己的游戏规则，否则很容易和评测逻辑不一致。

#### 6.2 AI 入口与打包

评测时统一入口是 [AI/main.py](./AI/main.py)。它只负责协议通信，并从打包后的 `ai.py` 中导入 `AI` 类。

当前仓库提供了这些打包脚本：

- [AI/zip_rand.sh](./AI/zip_rand.sh)
- [AI/zip_mcts.sh](./AI/zip_mcts.sh)
- [AI/zip_greedy.sh](./AI/zip_greedy.sh)
- [AI/zip_example.sh](./AI/zip_example.sh)

例如，打包 example AI：

```bash
bash AI/zip_example.sh
```

打包后的目录约定是：

- `main.py`：统一通信入口
- `ai.py`：当前参赛 AI 的公开入口
- `SDK/`：运行时依赖
- `tools/`：必要辅助脚本

如果你要新增自己的 AI，建议直接仿照 `AI/ai_example.py` 新建一个 `AI/ai_xxx.py`，再为它补一个对应的 `zip_xxx.sh`。

#### 6.3 训练脚本入口

SDK 根目录下公开了两个训练示例：

- [SDK/train_example.py](./SDK/train_example.py) / [SDK/train_example.sh](./SDK/train_example.sh)
- [SDK/train_mcts.py](./SDK/train_mcts.py) / [SDK/train_mcts.sh](./SDK/train_mcts.sh)

其中：

- `train_example`：最小示例，展示如何从 `SDK.backend` 和 `SDK.training` 接入框架
- `train_mcts`：MCTS 自对弈脚手架，展示如何收集对局并在后续加入参数更新逻辑

运行示例：

```bash
bash SDK/train_example.sh --seed 1 --max-actions 16
bash SDK/train_mcts.sh --episodes 2 --iterations 24 --max-depth 3 --seed 1
```

参考实现：

- 训练状态、环境、规则推进：使用 `SDK.backend` 和 `SDK.training`
- 特征、动作枚举等通用辅助：使用 `SDK.utils`
- 模型更新、损失函数、采样策略：写在你自己的训练脚本里
- AI 的决策逻辑：写在 `AI/` 中
